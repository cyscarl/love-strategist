反复加载完整聊天记录会快速消耗大量Token。建议采用 **“分层上下文 + 滑动窗口摘要”** 策略

│                      ~1000+ 条消息                      │
└─────────────────────────────────────────────────────────┘
                              │
                              ▼ (异步后台处理)

│   {性格: "外向", 爱好: ["摄影","旅行"], 好感度: 45}    │

└─────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────┐
│            近期对话窗口 (Sliding Window)               │

└─────────────────────────────────────────────────────────┘
                              │
                              ▼

│   Token: ~200-300                                      │

                              │

│   总 Token: ~1500-2500 (而非全部历史的上万Token)       │
└─────────────────────────────────────────────────────────┘





**实现要点：**

- **人物画像**：由智能体定期（或每次对话后）从聊天记录中提炼并更新，存储在SQLite中
- **滑动窗口**：默认取最近30条消息作为即时上下文
- **历史摘要**：对30条之前的对话，由LLM生成300字以内的摘要，随窗口滑动而更新
- **动态调整**：可根据对话长度和重要性动态调整窗口大小

**参考工具**：可关注 `clm-core` 等语义压缩库，能实现60-95%的Token压缩率
